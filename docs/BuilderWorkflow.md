# Builder Workflow

## Purpose

Builder Workflow는 AI Builder OS의 표준 개발 프로세스이다.

모든 프로젝트는 동일한 Workflow를 따르며, 프로젝트가 끝날 때마다 Workflow를 개선한다.

Workflow는 서비스보다 중요한 자산이다.

## Phase 1. Capture

## Goal

아이디어를 빠르게 기록한다.

Input

* 새로운 아이디어
* 사용자 요청
* 문제 발견
* 개선 사항

## Tasks

* Notion Service Backlog 등록
* Builder Backlog 등록
* 간단한 메모 작성

Output

* Backlog Item

---

## Phase 2. Evaluate

## Goal

아이디어의 우선순위를 판단한다.

Checklist

* 해결하려는 문제가 명확한가?
* AI 활용 가치가 있는가?
* MVP 제작이 가능한가?
* Builder OS에 도움이 되는가?
* 기존 Workflow를 재사용할 수 있는가?

Decision

Pass

↓

Research

Fail

↓

Archive

## Phase 11. Builder OS Update

프로젝트 종료 후 반드시 아래를 업데이트한다.

* Playbook
* Workflow
* Prompt Library
* Agent Rules
* Decision Log
* Documentation

Builder OS는 프로젝트가 끝날 때마다 더 발전해야 한다.

---

Exit Criteria

각 단계는 아래 조건을 만족해야 다음 단계로 이동한다.

Phase	Exit Criteria
Capture	Backlog 등록 완료
Evaluate	개발 여부 결정
Research	문제와 시장 정의 완료
Planning	PRD 승인
Design	핵심 화면 설계 완료
Development	MVP 동작
QA	Critical Bug 없음
Deploy	Production 배포 완료
Measure	KPI 수집 시작
Improve	다음 개선안 정의 완료

---

Workflow Rules

* 단계를 건너뛰지 않는다.
* MVP를 먼저 만든다.
* 반복 작업은 Workflow로 만든다.
* 같은 문제를 두 번 해결하지 않는다.
* Builder OS를 항상 함께 개선한다.
* 모든 산출물은 다음 프로젝트에서 재사용 가능해야 한다.

---

## Philosophy

서비스는 일회성 결과물이다.

Workflow는 반복 가능한 시스템이다.

Builder OS의 진짜 제품은 서비스가 아니라 Workflow이다.
