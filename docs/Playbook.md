# Playbook

## Purpose

Playbook은 AI Builder OS의 표준 운영 매뉴얼이다.

모든 프로젝트는 Playbook을 기준으로 진행하며, 예외가 발생하면 Decision Log에 기록하고 Playbook을 개선한다.

---

## Operating Principles

모든 프로젝트는 아래 원칙을 따른다.

* AI First
* MVP First
* Launch > Perfection
* System over Feature
* Workflow is Asset
* Automation by Default
* Reuse Before Build

## Step 1. Capture

Objective

아이디어를 잃지 않고 즉시 기록한다.

Checklist

* Service Backlog 등록
* Builder Backlog 등록
* 한 줄 설명 작성

---

## Step 2. Evaluate

Objective

개발 여부를 판단한다.

Checklist

* 실제 문제인가?
* AI가 해결할 수 있는가?
* 2주 안에 MVP 가능한가?
* 기존 Workflow 재사용 가능한가?
* Builder OS 개선에 기여하는가?

Decision

Pass → Research

Fail → Archive

---

## Step 3. Research

Objective

문제를 이해한다.

Checklist

* 경쟁 서비스 조사
* 사용자 조사
* 시장 조사
* 차별화 포인트 정의

---

## Step 4. Planning

Objective

개발 가능한 수준으로 정의한다.

## Deliverables

* PRD
* User Story
* Acceptance Criteria
* MVP Scope

---

## Step 5. Design

Objective

최소한의 UI와 UX를 설계한다.

Checklist

* User Flow
* IA
* Wireframe
* Responsive
* Design System 적용

---

## Step 6. Development

Objective

동작하는 MVP를 만든다.

## Rules

* 기능보다 완료를 우선한다.
* 작은 단위로 구현한다.
* 재사용 가능한 구조를 만든다.

---

## Step 7. QA

Objective

출시 가능한 수준인지 확인한다.

Checklist

* 기능 테스트
* Edge Case
* 모바일 테스트
* 성능
* SEO
* 접근성

---

## Step 8. Deploy

Objective

실제 사용 가능한 서비스를 배포한다.

Checklist

* Production Deploy
* Analytics 연결
* Error Tracking
* Monitoring

---

## Step 9. Measure

Objective

데이터를 기반으로 판단한다.

## Metrics

* Traffic
* Activation
* Retention
* Conversion
* Revenue
* Error Rate

---

## Step 10. Improve

Objective

서비스보다 Builder OS를 먼저 개선한다.

Checklist

* Prompt 개선
* Workflow 개선
* Automation 추가
* Agent 개선
* Documentation 업데이트

---

## Decision Rules

Two Week Rule

2주 이상 걸리면 MVP 범위를 줄인다.

---

AI First Rule

AI가 할 수 있는 일은 사람이 하지 않는다.

---

Workflow Rule

같은 작업을 두 번 수행하면 Workflow로 만든다.

---

Automation Rule

반복 작업은 반드시 자동화 후보로 등록한다.

---

Reuse Rule

새로 만들기 전에 기존 Prompt, Workflow, Template을 먼저 확인한다.

---

Launch Rule

핵심 기능이 동작하면 출시한다.

완벽한 첫 버전은 목표가 아니다.

---

Stop Rule

사용자 검증 없이 기능만 계속 추가하지 않는다.

---

## Definition of Done

프로젝트는 아래 조건을 만족하면 완료로 간주한다.

* MVP 출시
* QA 완료
* Analytics 연결
* Documentation 완료
* Builder OS 업데이트 완료
* Decision Log 작성

---

## Daily Routine

매일 아래 순서를 반복한다.

1. Backlog 확인
2. Sprint 확인
3. AI와 작업
4. Commit
5. Documentation
6. Retrospective

---

## Continuous Improvement

프로젝트 종료 후 반드시 아래를 개선한다.

* Playbook
* Workflow
* Prompt Library
* Agent Rules
* Decision Log

Builder OS는 프로젝트가 끝날 때마다 더 강력해져야 한다.

---

## Philosophy

좋은 서비스보다

좋은 Workflow가 더 오래 살아남는다.

Builder OS의 가장 중요한 제품은 서비스가 아니라 시스템이다.
